Il dominio di **ingestione** prende l'output dell'estrazione (IR StyledGrid, testo narrativo) e lo scrive
negli [store](/it/docs/guides/storage). Ha tre corsie: **strutturata** (fatti
numerici), **narrativa** (chunk di documenti) e **revisione** (la coda human-in-the-loop che
intercetta tutto ci├▓ di cui il percorso deterministico non ├© sicuro). Ogni corsia ├© idempotente ŌĆö
rieseguire un'ingestione non deve produrre scritture duplicate.

<Callout type="info">
  Pacchetto: `src/ragspine/ingestion/`. Contratto: `src/ragspine/ingestion/CLAUDE.md`.
</Callout>

## Ingestione strutturata

`structured/ingestion.py` orchestra un documento dall'inizio alla fine:
**estrazione ŌåÆ normalizzazione (glossario) ŌåÆ tag colore ŌåÆ upsert**. Ci sono due punti di ingresso:

- `ingest_excel(path, store, registry, queue, *, dry_run=False, extractor_version="xlsx_styled@1", manifest=None, batch_id=None)` ŌĆö
  solo xlsx.
- `ingest_file(path, store, registry, queue, *, dry_run=False, manifest=None, batch_id=None, valid_as_of=None, grid_extractor=None)` ŌĆö
  il dispatcher multi-formato unificato: instrada in base al suffisso verso l'estrattore corretto (xlsx /
  xlsm / pptx, oppure PDF tramite il [router](/it/docs/guides/extraction#instradamento-pdf--triage-per-pagina))
  e riutilizza la logica di ingestione condivisa.

Entrambi restituiscono un **`IngestReport`** (un oggetto di conteggi, non numeri grezzi):

<TypeTable
  type={{
    source_path: { type: 'str' },
    source_doc_id: { type: 'str' },
    file_hash: { type: 'str | None', default: 'None' },
    dry_run: { type: 'bool', default: 'False' },
    n_grids: { type: 'int', default: '0' },
    n_facts_extracted: { type: 'int', default: '0' },
    n_facts_ingested: { type: 'int', default: '0' },
    n_tags_applied: { type: 'int', default: '0' },
    n_enqueued_review: { type: 'int', default: '0' },
    warnings: { type: 'list[str]', default: '[]' },
    status: { type: 'str', default: '"ok"' },
    error: { type: 'str | None', default: 'None' },
  }}
/>

Internamente, a ogni griglia vengono applicati i tag colore tramite la mappatura attiva (`apply_mapping`), quindi viene trasformata in
oggetti [`Fact`](/it/docs/guides/storage#la-dataclass-fact) e scritta con
`store.upsert_facts(...)`. I fatti vengono marcati con la loro lineage ŌĆö
`source_doc_id`, `source_locator`, `source_file_hash`, `extractor_version`,
`mapping_version` e `review_status=REVIEW_AUTO_APPROVED`.

```python
from ragspine.ingestion.structured.ingestion import ingest_file

report = ingest_file("report.xlsx", store, registry, queue)
print(report.n_facts_ingested, report.n_enqueued_review)
```

### Quando l'ingestione mette invece in coda per la revisione

Il percorso strutturato ├© conservativo. Un file viene instradato verso la [coda di revisione](#la-coda-di-revisione)
invece di essere ingerito automaticamente quando:

- nessuna griglia riesce a risolvere un'entit├Ā eppure il file contiene dati estraibili ŌĆö motivo
  _"Õ«×õĮōµŚĀµ│ĢĶ¦Żµ×É’╝īķ£Ćõ║║ÕĘźµīćĶ«ż"_ (entit├Ā non risolvibile, richiede identificazione umana);
- il file ha celle colorate ma lo scope **non ha alcuna mappatura colore attiva** ŌĆö motivo
  _"ķó£Ķē▓µśĀÕ░äµ£¬ńĪ«Ķ«ż’╝īķ£Ć SME ńĪ«Ķ«żÕøŠõŠŗ"_ (mappatura colore non confermata, lo SME deve confermare la legenda);
- un PDF sembra un'esportazione di PowerPoint (richiedere il file pptx sorgente) oppure ├© una scansione che necessita di OCR.

Con `dry_run=True`, l'estrazione e la reportistica vengono eseguite per intero ma `n_facts_ingested` e
`n_enqueued_review` restano `0` ŌĆö lo store e la coda rimangono intatti.

## Il registro manifest dei batch

`structured/ingestion_manifest.py` registra _cosa ├© stato eseguito_. `ManifestStore` (sqlite,
tabelle `manifest_batch` + `manifest_input`) apre un batch, registra ogni file di input e chiude
il batch con uno stato finale e una durata. Ogni batch ├© un `ManifestRecord`:

| campo                                 | significato                                           |
| ------------------------------------- | ------------------------------------------------- |
| `batch_id`                            | fornito dal chiamante, oppure auto `batch-{uuid4 hex[:12]}` |
| `status`                              | `running` ŌåÆ `done` / `failed`                     |
| `inputs`                              | righe `{path, hash, format, ŌĆ”}` per file           |
| `n_facts` ┬Ę `n_warnings` ┬Ę `n_failed` | conteggi aggregati                                  |
| `duration_s` ┬Ę `failures`             | tempistiche + errori per file                          |

API: `open_batch(batch_id=None)`, `record_input(...)`, `close_batch(batch_id, status="done")`,
`get_batch(id)`, `list_batches()`. Due helper di osservabilit├Ā li accompagnano:
`compute_metrics(manifest_store, queue, store)` (totali dei fatti, arretrato di revisione, fasce di
confidenza, tasso di warning) e `list_versions(store, registry)` (versioni attive degli estrattori +
mappature colore).

<Callout type="warn">
  **Dove risiede davvero l'idempotenza.** Il contratto definisce il manifest "la guardia", ed ├© il
  registro di audit di ogni esecuzione (percorso / hash / conteggi / fallimenti). Ma la garanzia letterale di assenza di
  scritture duplicate proviene dall'**upsert a chiave univoca** dello store dei fatti (`store.upsert_facts`, con chiave su
  [`dim_key`](/it/docs/guides/storage#dim_key--la-chiave-di-upsert)): rieseguire un batch riesegue estrazione e
  upsert, e la chiave univoca impedisce allo store di crescere. `batch_id` **non** ├© derivato dal contenuto
  ŌĆö ├© fornito dal chiamante oppure ├© un uuid casuale.
</Callout>

## Ingestione narrativa

La corsia narrativa ├© composta da due moduli con una separazione netta:

<Tabs items={["narrative_extract", "narrative_ingest"]}>
<Tab value="narrative_extract">

Estrazione di testo pura e deterministica ŌĆö zero OCR, zero LLM, nessuno store. `extract_narrative(path)`
smista in base al suffisso (`SUPPORTED_SUFFIXES = {.pptx, .pdf, .docx, .docm, .txt}`) verso l'estrattore
corrispondente e restituisce un **`NarrativeDoc`**: `doc_id`, `file_hash`, un elenco di `NarrativeSegment`
(`text` + `source_locator`), `skipped_pages` e `warnings`. I locator hanno l'aspetto di
`'slide={N},frame={M}'`, `'slide={N},notes'`, `'page={N}'`, oppure (per il testo semplice) `'para={N}'`.
`NarrativeDoc.to_text()` unisce i segmenti con righe vuote ŌĆö quella stringa ├© il contratto di input
del chunking.

I file `.txt` semplici seguono lo stesso percorso narrativo ŌĆö trattati come prosa continua, mai forzati
in fatti strutturati. `extract_txt_narrative(path)` legge UTF-8 (con `errors="replace"`),
divide sulle righe vuote in blocchi di paragrafi, normalizza ogni blocco ed emette un
`NarrativeSegment` per ogni blocco non vuoto con un `source_locator="para={N}"` a base 1. ├ł
privo di dipendenze e deterministico.

</Tab>
<Tab value="narrative_ingest">

Orchestrazione batch: **estrazione ŌåÆ chunking ŌåÆ scrittura nello store dei chunk**, idempotente ed eseguibile in dry-run.
`ingest_narrative(inputs, store, *, meta_by_doc=None, dry_run=False, chunker=None)` accetta una
cartella, un file o un elenco, e restituisce un `NarrativeIngestReport` (un elenco di
`FileReport` per file, pi├╣ `counts()`). Con `chunker=None`, ogni file usa
`chunk_document(doc.to_text(), doc_meta)`; un `Chunker` iniettato pu├▓ selezionare il comportamento parent-child,
sentence-window, layout, dominio o semantico. I campi di gerarchia/finestra risultanti vengono
persistiti nello store dei chunk invece di essere ricostruiti solo in memoria. I chunk vengono
scritti con `store.replace_doc_chunks(...)` nel
[ChunkStore](/it/docs/guides/storage#larchivio-di-chunk).

Lo stato per file ├© uno tra `ingested` / `skipped` / `failed` / `no_text`. L'idempotenza usa una
tabella `narrative_doc` (`doc_id ŌåÆ file_hash`) nello stesso DB sqlite: se l'hash registrato
corrisponde al file, il file viene **saltato** senza rieseguire l'estrazione. Le chiavi di `meta_by_doc` sono
validate rispetto a `ALLOWED_META_KEYS` (`title`, `topic`, `entity`, `geography`, `period`,
`language`, `sensitivity`, `valid_as_of`) ŌĆö i campi sconosciuti sollevano `ValueError`. Il periodo ├©
preso dai metadati oppure dedotto dal nome del file tramite `period_from_filename(name)`.

</Tab>
</Tabs>

<Callout type="info">
  La sensibilit├Ā viene applicata qui: un `meta["sensitivity"]` esplicito ha la precedenza, altrimenti
  viene eseguito `classify_sensitivity(...)` da [common](/it/docs/guides/common#sensibilit├Ā). ├ł questo che
  in seguito consente al recupero di applicare l'isolamento RESTRICTED.
</Callout>

## La coda di revisione

`review/review_queue.py` ├© la **macchina a stati** per la revisione umana degli SME che intercetta tutto
ci├▓ di cui il percorso deterministico non ├© sicuro ŌĆö OCR a bassa confidenza, conflitti tra canali,
mappature colore non confermate, entit├Ā non risolvibili. ├ł basata su sqlite (stesso DB dello store dei
fatti, tabelle diverse: `review_item` + un `review_audit` append-only).

La macchina a stati ha tre stati stringa e due transizioni:

```text
pending ŌöĆŌöĆapproveŌöĆŌöĆŌ¢Č approved   (terminal)
pending ŌöĆŌöĆrejectŌöĆŌöĆŌöĆŌ¢Č rejected   (terminal)
```

`STATUS_PENDING = "pending"`, `STATUS_APPROVED = "approved"`, `STATUS_REJECTED = "rejected"`.
Gli stati approvato e rifiutato sono terminali ŌĆö rielaborare un elemento terminale (o agire su un
elemento inesistente) solleva `IllegalTransitionError`.

Un **`ReviewItem`** contiene `reason`, `payload` (JSON), `locator`, `priority` (predefinito `100`,
**pi├╣ basso = revisionato prima**), `id`, `status`, `actor`, `note` e `corrected_value`.

API:

| metodo                                                    | effetto                                               |
| --------------------------------------------------------- | ---------------------------------------------------- |
| `enqueue(reason, payload, locator, priority=100) -> int`  | inserisce un elemento in attesa + scrive una riga di audit `enqueue` |
| `list_pending() -> list[ReviewItem]`                      | elementi in attesa, ordinati per `priority ASC, id ASC`        |
| `approve(item_id, actor, note=None)`                      | ŌåÆ approvato                                           |
| `reject(item_id, actor, note=None, corrected_value=None)` | ŌåÆ rifiutato (opzionalmente registra una correzione)          |
| `get(item_id)` ┬Ę `audit_trail(item_id)`                   | recupera l'elemento / cronologia `AuditRecord` append-only       |

Ogni transizione appende un `AuditRecord` (`enqueue` / `approve` / `reject`) ŌĆö il tracciato ├©
append-only e non viene mai modificato, quindi la cronologia delle revisioni ├© completamente ricostruibile. Vedi
[Coda di revisione nel glossario](/it/docs/concepts/glossary#coda-di-revisione).

## Connettori sorgente

Entrambe le corsie ingeriscono da percorsi locali per impostazione predefinita, ma la sorgente ├© essa stessa un punto di estensione pluggabile.
`ingestion/source/connector.py` definisce il Protocol `SourceConnector` ŌĆö un singolo
`iter_documents() -> Iterable[RawDoc]` che restituisce ogni documento come un `RawDoc` frozen
(`source_doc_id`, `locator`, `content: bytes`, `content_type`, `metadata`).

Il `FilesystemConnector` predefinito ├© privo di dipendenze. `InMemoryConnector` fornisce documenti
in-process (test / fixture). `HttpConnector` e `NotionConnector` raggiungono knowledge
base remote e importano in modo lazy `httpx` dietro l'extra `[connectors]`. Sceglierne uno con
`make_source_connector(spec, **kwargs)` o `RAGSPINE_SOURCE_CONNECTOR`; i connettori di terze parti si
registrano tramite il gruppo di entry-point `ragspine.source_connectors` (in caso di
conflitto vincono i nomi built-in). Vedi [Punti di estensione ŌåÆ SourceConnector](/it/docs/reference/extension-points#sourceconnector).

```python
from ragspine.ingestion.source.connector import make_source_connector

# None / "none" ŌåÆ None; "filesystem"/"fs" ŌåÆ local walk; "http"/"notion" ŌåÆ remote ([connectors]).
connector = make_source_connector("filesystem")
for raw in connector.iter_documents():
    ...  # raw is a RawDoc: source_doc_id, locator, content bytes, content_type, metadata
```

<Callout type="info">
  L'extra `[connectors]` installa soltanto `httpx` (con licenza permissiva, importato in modo lazy). Senza alcun
  connettore selezionato, l'ingestione legge i percorsi locali esattamente come prima ŌĆö identica byte per byte.
</Callout>

## Invarianti garantite da questo dominio

- **Ingestione idempotente** ŌĆö le riesecuzioni strutturate eseguono upsert su `dim_key`; le riesecuzioni narrative saltano
  in caso di `file_hash` corrispondente. Rieseguire l'ingestione non raddoppia mai lo store.
- **Provenienza preservata** ŌĆö ogni fatto e ogni chunk mantiene il proprio `source_doc_id` + locator.
- **Auto-ingestione conservativa** ŌĆö tutto ci├▓ che ├© ambiguo (entit├Ā / mappatura / confidenza /
  conflitto) va a un essere umano, non silenziosamente nello store.
- **Audit append-only** ŌĆö le transizioni di revisione vengono registrate, mai sovrascritte.
